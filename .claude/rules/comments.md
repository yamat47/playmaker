# コメント規約

最優先価値は **商用ソフトへの組み込みやすさ**。コメントもその一部であり、
**読み手が最短で意図に到達できること**を目的とする。経緯の真実源は git 履歴・
`docs/plans/` ・memory であり、コードはそこに従属しない。

## 核心原則（t-wada / 現代的知見）

- **コードで語れることはコードで語る** — 命名・関数分割・型で表現できるものを
  コメントで補わない。コメントが必要なコードは設計の改善余地のサイン
- **コメントは風化し、嘘をつく** — コードと乖離するため、量より「腐らない価値」
  を優先する。更新できない／古くなった記述は書かない・残さない
- **経緯・履歴はコメントに残さない** — 実装の過程・進め方は VCS / 設計ドキュメント
  / memory に置く。コードに **プロセスの residue を残さない**

## 書く（KEEP）

1. **Why** — 非自明な判断・制約・トレードオフを **なぜ** そう選んだか
2. **Why-not** — 検討して却下した代替、避けた罠（同じ失敗の再発防止）
3. **契約・不変条件** — 型で表せない事前/事後条件、呼び出し側が守るべき責務
4. **警告・落とし穴** — 非自明な副作用・順序依存・パフォーマンス上の理由
5. **安定した外部仕様への参照** — PRD の章番号など、長期に存在し Why を補強する
   もの。**ただし Why を補強せず装飾的なだけの参照タグは省く**
6. **lint / カバレッジ抑制の理由** — `biome-ignore` / `v8 ignore` は Why 必須

## 書かない（REMOVE）

1. **What** — コードを読めば分かる説明（命名・分割で解消する）
2. **実装過程・マイルストーン参照** — `M0`〜`M9`、「〜で確定」「〜へ繰り延べ」
   「（M4 契約）」のような進め方の痕跡。寿命の短いプロセスはコードに残さない。
   経緯は git 履歴・`docs/plans/implementation-roadmap.md` ・memory が真実源
3. **コメントアウトされた死コード** — git が履歴を持つので消す
4. **古い／更新されない記述** — 嘘になる前に消すか、コードを直す
5. **Why を持たない純装飾の区切り** — `// --- xxx ---` の多用は避け、構造は
   ファイル分割・関数で表現する（節見出しは必要最小限に留める）

## TODO / FIXME

コードに TODO を残さない。やり残し・繰り延べは `docs/plans/` と memory に集約する
（TODO はレビューされず腐る）。どうしてもコードに書く必要があるなら Issue 番号 /
期限 / 理由を必ず添える。

## 判断に迷ったら

「この一文は **半年後にコードだけ読む人** が必要とするか、それとも当時の進め方を
知っている人向けの memo か」で切る。後者なら消す（git / docs が引き受ける）。
PRD 参照は **その制約がなぜ存在するか** を説明しているなら残し、ラベルとして
貼ってあるだけなら消す。
