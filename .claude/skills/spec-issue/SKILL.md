# GitHub Issue 仕様策定

GitHub Issue $ARGUMENTS の仕様を対話的に明確化します。

## Step 1: Issue の確認

```bash
gh issue view $ARGUMENTS
```

Issue の内容を読み、概要を把握する。

## Step 2: 現状の実装を調査

Explore agent を使って、関連する現状の実装を調査する。
- 関連するモデル、コントローラー、ビュー
- 既存のパターンや規約
- 再利用できそうなコード

## Step 3: 対話的に仕様を明確化

**AskUserQuestion を使って質問する。**
以下の7カテゴリを順に確認していく。

### 3.1 目的・背景（Why）

なぜこの機能が必要かを明確にする。

- この機能で解決したい課題は何ですか？
- なぜ今この機能が必要ですか？
- この機能がないと誰がどう困りますか？
- 成功の定義は何ですか？（KPI/メトリクス）

### 3.2 ユーザー・ステークホルダー（Who）

誰が使うか、誰が関係するかを明確にする。

- この機能を使うのは誰ですか？
- 意思決定者（最終承認者）は誰ですか？
- 影響を受ける他のチーム・システムはありますか？

### 3.3 スコープ・境界（What）

含めるもの・含めないものを明確にする。

- この機能に含まれるもの・含まれないものは？
- MVP（最小限の機能）は何ですか？
- 将来の拡張予定はありますか？（今は含めない）

### 3.4 導線・ユーザーフロー（How）

正常系の操作フローを明確にする。

- ユーザーはどこからこの機能にアクセスしますか？
- URL構造はどうしますか？
- 主要な操作フローを教えてください
- 表示する情報は何ですか？
- 完了後、ユーザーはどこに遷移しますか？

### 3.5 エッジケース・エラー処理（What if）

「もし〜だったら」を洗い出す。

- データが存在しない場合はどうしますか？
- 無効な入力があった場合の挙動は？
- 権限がないユーザーがアクセスした場合は？
- 処理中にエラーが発生した場合は？
- 同時に複数人が操作した場合は？

### 3.6 非機能要件（How much / How secure）

品質特性を確認する。

- 認証は必要ですか？（ログイン必須/不要）
- パフォーマンス要件はありますか？（レスポンス時間等）
- セキュリティ要件は？（認可、データ保護）
- 既存システムとの連携は？

### 3.7 受け入れ基準・検証方法（Done）

完了条件を明確にする。

- どうなれば「完了」と言えますか？
- 受け入れテストの方法は？
- ステークホルダーの承認プロセスは？

## 質問のコツ

- **選択肢を提示する**: 「AとBどちらですか？」形式が答えやすい
- **具体例を出す**: 「例えば /companies/:id のような形式ですか？」
- **仮説を立てる**: 「〜という理解で合っていますか？」
- **深掘りする**: 回答が曖昧な場合は追加で質問する

## Step 4: 仕様をまとめる

質問が終わったら、以下の形式で仕様をまとめる。

```markdown
## 仕様サマリー

### 概要
[1-2文で機能の説明]

### 目的・背景
[解決したい課題、成功の定義]

### ユーザー
[誰が使うか]

### URL/導線
[アクセス方法、URL構造]

### 表示内容
[画面に表示する情報]

### 動作仕様
| 条件 | 挙動 |
|------|------|
| 正常系 | ... |
| データなし | ... |
| エラー時 | ... |
| 権限なし | ... |

### 認証/権限
[認証要件]

### スコープ外
[今回含めないもの]

### 受け入れ基準
[完了条件]
```

## Step 5: ユーザーに確認

まとめた仕様をユーザーに見せて、認識にズレがないか確認する。

## Step 6: GitHub Issue にコメント（オプション）

ユーザーが希望すれば、仕様を GitHub Issue にコメントとして追加する。

```bash
gh issue comment $ARGUMENTS --body "..."
```

## 注意事項

- **必要な質問は省略しない**: 曖昧なまま進めると後で手戻りになる
- **早めにまとめる**: 8割固まったらまとめて、残りは確認で埋める
- **実装の詳細は後回し**: 「何を作るか」を決める、「どう作るか」は実装時に
