# docs/prd.md を全面見直しするプラン

## Context

`docs/prd.md` を全面的に見直す。現 PRD は 2025年12月に「npm パッケージ化 + Rails 統合」方針で刷新されたが、
その後ソースコード一式が削除され、現在はコードが存在しない「再スタート直前」の状態。
また、その後ユーザーの方針が変化しており（日本国内向け・商用化視野・商用ソフトは別リポジトリ・クローズド維持）、
現 PRD の前提（`private: true` で npm 非公開 / UI フレームワーク非依存の汎用ライブラリ / Hudl を参考）と
新方針のあいだに矛盾や未整理の箇所がある。

grill-me プロセスで根本から決定を積み上げ、新しい PRD の方向性を以下に固めた。
次のフェーズでこの合意に基づき `docs/prd.md` を全面書き換える。

## 合意した設計（ツリー構造）

### PRD の性質
- **Why と What に絞る**（技術仕様は別文書）
  - **ライブラリ単体の PRD** として書く
    - 背景セクションで商用ソフトとの関係に軽く触れる
    - 商用ソフト側の要件（チーム管理・課金・共有等）はこの PRD では扱わない

### プロジェクトの立ち位置
- 日本国内のアメフト関係者向け、**商用化も視野**
  - このリポジトリの範囲: **プレー図作成ライブラリのみ**（JavaScript ライブラリ的にアウトプット）
    - 商用ソフトウェアは**別リポジトリ**で開発
  - **MVP 前提** で最初のリリース可能な版を作る

### ライブラリの最優先価値
- **商用ソフトへの組み込みやすさ**（エンドユーザー体験・出力品質・正確性より優先）
  - 提供形: **UI + ロジック一体**
    - 見た目: **CSS 変数**でカスタマイズ
    - 振る舞い: **`onChange` 等のフック**で外部制御
    - 商用ソフト側は「薄いラッパー + データ永続化」だけ書けば済む状態を目指す
  - 言語: **日本語デフォルトのみ**（i18n 機構なし、シンプル優先）
  - プラットフォーム: **PC + マウス専用**（タッチ動作は保証しない）
  - 公開: **クローズド / private 維持**（商用ソフト専用部品、OSS 化予定なし）

### MVP のスコープ
- MVP に含める機能: **現 PRD の全仕様**
  - 選手配置（6 形状、ラベル・色編集、複数人）
  - 線描画 3 種（route / block / motion）+ 直線・ベジェ曲線
  - 3 ゾーン切替（middle / redzone / own-redzone）
  - Undo/Redo、プロパティ編集
  - フォーメーションテンプレート読み込み
  - PNG エクスポート
  - view / edit モード切替
  - `onChange` コールバックと PlayData 入出力
- MVP 完了基準: **現 PRD の全仕様が動作 + テスト通過**（機能カバレッジ基準）
  - 実使用での検証は商用ソフト側の責務

### 残り論点（まとめて合意済み）
- **対象外に追加**: モバイル/タッチ操作、i18n、OSS 化準備ドキュメント
- **技術仕様の扱い**: 現 PRD の API 署名・型定義・Z オーダー・エラー詳細は PRD から削除。Tech spec は実装時に別途作成（今は作らない）
- **ベンチマーク**: 現 PRD 通り Hudl 参考のまま
- **想定ユーザー**: 一次ユーザー（商用ソフト開発者 = 作り手）と二次ユーザー（日本のアメフト指導者）を明示
- **非機能要件**: ブラウザ・パフォーマンス・UI フレームワーク非依存・スタイルカスタマイズ・データバージョニングのみ残す

## 新しい `docs/prd.md` の章立て

1. **背景**
   - なぜ作るか（日本のアメフト指導者向けの商用ソフトを作るにあたり、図作成コアを独立したライブラリとして切り出す）
   - 商用ソフトとの関係（商用ソフトは別リポジトリ、このライブラリはそこに組み込まれるコンポーネント）
2. **スコープ**
   - このリポジトリは「プレー図作成ライブラリ」のみを扱う
   - 責務に含むもの: プレー図の描画・編集 UI・`PlayData` の入出力
   - 責務に含まないもの（商用ソフト側の責務）: 永続化、認証、チーム管理、共有、プレーブック全体管理
3. **想定ユーザー**
   - **一次ユーザー**: 商用ソフト開発者（= 作り手自身）。「組み込みやすさ」の評価者
   - **二次ユーザー**: 日本のアメフト指導者・コーチ。商用ソフト経由でこのライブラリに触れる
4. **大切にする価値**
   - 最優先: **商用ソフトへの組み込みやすさ**
     - 副次: UI フレームワーク非依存、CSS 変数でカスタマイズ、onChange フックで振る舞い制御、データ可搬性
   - 明示的に最優先にしないもの: エンドユーザー体験の最適化（商用ソフト側で調整する前提）、出力品質の極大化、アメフト戦術表現の厳密性
5. **機能要件（概要レベル）**
   - フィールド描画（ハーフフィールド、3 ゾーン切替）
   - 選手配置（6 形状、プロパティ編集、Add/Delete/Drag）
   - 線描画 3 種（route / block / motion、直線・ベジェ曲線、waypoint 編集）
   - edit モード（ツールバー + プロパティパネル + Undo/Redo）
   - view モード（読み取り専用表示）
   - フォーメーションテンプレート（組み込み + 外部読み込み）
   - PNG エクスポート
   - `onChange` による PlayData 通知
   - ※ API 署名・型定義・Z オーダー・バリデーション詳細は別文書（Tech spec）に分離
6. **非機能要件**
   - ブラウザサポート: モダンブラウザ（Chrome/Firefox/Safari/Edge の最新 2 バージョン、デスクトップのみ）
   - パフォーマンス: 選手 22 人 + 線 20 本程度で遅延なく操作できる
   - UI フレームワーク非依存（DOM API スタイルで提供）
   - ランタイム依存最小
   - CSS 変数によるスタイルカスタマイズ
   - データバージョニング（`PlayData.version` フィールドで将来マイグレーションを想定）
7. **成功基準**
   - 上記機能要件のすべてが仕様通り動作する
   - ユニットテスト・統合テストが通る
   - 実使用での検証は商用ソフトの PRD の責務とし、ここでは扱わない
8. **対象外**
   - アニメーション（プレー再生）
   - PDF / SVG エクスポート（PNG のみ）
   - 動画連携
   - データの永続化・認証・チーム管理・共有
   - **モバイル専用 UI / タッチ操作サポート**（追加）
   - LOS 描画
   - マルチセレクト
   - ズーム / パン
   - コピー & ペースト
   - Undo/Redo 以外のキーボードショートカット
   - アクセシビリティ対応
   - コンテナサイズの最小値・最大値制約
   - **多言語対応（i18n）**（追加）
   - **OSS 公開を前提とした汎用ドキュメント・ライセンス整備**（追加）
